# Plugin System

The kustomark plugin system allows you to extend markdown processing with custom JavaScript/TypeScript transformations. Plugins can modify content, add metadata, validate links, generate tables of contents, and much more.

## Overview

Plugins in kustomark are JavaScript/TypeScript modules that transform markdown content. They run through the `exec` patch operation, which applies plugin transformations to matched files.

### Key Features

- **Simple Interface**: Plugins export a small set of required fields
- **TypeScript Support**: Full type safety with TypeScript
- **Async Support**: Plugins can be synchronous or asynchronous
- **Parameter Validation**: Built-in parameter validation system
- **Timeout Protection**: Automatic timeout enforcement prevents hanging
- **Checksum Verification**: Optional checksums for security
- **Context Access**: Access to file path, config, and safe environment variables

### When to Use Plugins

Use plugins when you need to:

- Generate dynamic content (TOC, statistics, badges)
- Validate or transform content programmatically
- Add metadata to frontmatter
- Format code blocks or tables
- Integrate with external tools
- Perform complex transformations beyond regex

## Security Model

The kustomark plugin system includes several security features:

### Sandboxing

- **Environment Filtering**: Sensitive environment variables (passwords, tokens, keys) are automatically filtered out
- **Readonly Config**: Plugin context provides readonly access to configuration
- **Timeout Enforcement**: Plugins are automatically terminated if they exceed the timeout
- **Checksum Verification**: Optional SHA256 checksums ensure plugin integrity

### Trust Model

Plugins run with the same permissions as the kustomark process. Only use plugins from trusted sources.

- Local plugins: Full file system access
- npm plugins: Runs with npm package permissions
- Always review plugin source code before use

### Checksum Verification

Add checksums to prevent unauthorized plugin modifications:

```yaml
plugins:
  - name: toc-generator
    source: ./plugins/toc.js
    checksum: sha256:abc123def456...
```

Generate checksums with:

```bash
sha256sum plugins/toc.js
```

## Quick Start

### 1. Create a Simple Plugin

Create `plugins/footer.js`:

```javascript
export const name = 'footer';
export const version = '1.0.0';

export const apply = (content, params, context) => {
  const text = params.text || 'Generated by kustomark';
  return `${content}\n\n---\n\n${text}`;
};
```

### 2. Configure Plugin

Add to `kustomark.yaml`:

```yaml
apiVersion: kustomark/v1
kind: Kustomization
resources:
  - "docs/**/*.md"

plugins:
  - name: footer
    source: ./plugins/footer.js

patches:
  - op: exec
    plugin: footer
    params:
      text: "Last updated: 2024"
```

### 3. Run kustomark

```bash
kustomark build
```

The plugin will add a footer to all matched markdown files.

## Plugin Authoring Guide

### Required Exports

Every plugin must export:

```javascript
// Unique plugin identifier
export const name = 'my-plugin';

// Semantic version
export const version = '1.0.0';

// Main transformation function
export const apply = (content, params, context) => {
  // Transform content and return result
  return content;
};
```

### Optional Exports

```javascript
// Plugin description
export const description = 'What this plugin does';

// Parameter validation function
export const validate = (params) => {
  const errors = [];
  // Validate params and return errors
  return errors;
};

// Parameter schema (for documentation)
export const params = [
  {
    name: 'maxDepth',
    type: 'number',
    required: false,
    default: 3,
    description: 'Maximum depth for processing'
  }
];
```

### The Apply Function

The `apply` function signature:

```typescript
type PluginFunction = (
  content: string,        // Current markdown content
  params: Record<string, unknown>,  // User parameters
  context: PluginContext  // Execution context
) => string | Promise<string>;  // Transformed content
```

#### Context Object

```typescript
interface PluginContext {
  file: string;           // Absolute file path
  config: Readonly<KustomarkConfig>;  // Kustomark config
  env: Readonly<Record<string, string>>;  // Safe env vars
  relativePath: string;   // Relative path from output
}
```

Example usage:

```javascript
export const apply = (content, params, context) => {
  console.log(`Processing: ${context.file}`);
  console.log(`Output path: ${context.relativePath}`);

  return content;
};
```

### Parameter Validation

Implement the `validate` function to check parameters before execution:

```javascript
export const validate = (params) => {
  const errors = [];

  if (params.maxDepth !== undefined) {
    if (typeof params.maxDepth !== 'number') {
      errors.push({
        param: 'maxDepth',
        message: 'maxDepth must be a number',
        expected: 'number',
        actual: typeof params.maxDepth
      });
    }

    if (params.maxDepth < 1 || params.maxDepth > 6) {
      errors.push({
        param: 'maxDepth',
        message: 'maxDepth must be between 1 and 6',
        expected: '1-6',
        actual: String(params.maxDepth)
      });
    }
  }

  return errors;
};
```

If validation returns any errors, the plugin will not execute.

### Async Plugins

Plugins can be asynchronous:

```javascript
export const apply = async (content, params, context) => {
  // Simulate async operation
  await new Promise(resolve => setTimeout(resolve, 100));

  // Can use await with async operations
  const result = await fetchExternalData();

  return `${content}\n\n${result}`;
};
```

### Error Handling

Throw errors for unrecoverable conditions:

```javascript
export const apply = (content, params, context) => {
  if (!params.required) {
    throw new Error('Missing required parameter: required');
  }

  return content;
};
```

Errors will be caught and reported with context:

```
Plugin 'my-plugin' failed while processing '/project/docs/file.md': Missing required parameter: required
```

## Configuration

### Plugin Declaration

Declare plugins in `kustomark.yaml`:

```yaml
plugins:
  - name: plugin-name          # Unique identifier
    source: ./path/to/plugin.js  # File path or npm package
    version: ^1.0.0            # Optional: npm version constraint
    checksum: sha256:abc123...  # Optional: security checksum
    timeout: 30000             # Optional: plugin-specific timeout (ms)
```

### Source Types

#### Local File

```yaml
plugins:
  - name: my-plugin
    source: ./plugins/my-plugin.js  # Relative to kustomark.yaml
```

```yaml
plugins:
  - name: my-plugin
    source: /absolute/path/to/plugin.js  # Absolute path
```

#### npm Package

```yaml
plugins:
  - name: toc-generator
    source: kustomark-plugin-toc  # npm package
    version: ^1.0.0
```

```yaml
plugins:
  - name: analytics
    source: '@myorg/kustomark-plugin-analytics'  # Scoped package
    version: ~2.1.0
```

### Using Plugins in Patches

Use the `exec` operation to apply plugins:

```yaml
patches:
  - op: exec
    plugin: toc-generator
    params:
      maxDepth: 3
      title: "Contents"
```

With selectors:

```yaml
patches:
  - op: exec
    plugin: word-counter
    selector:
      files:
        - "docs/**/*.md"
    params:
      showReadingTime: true
```

With conditions:

```yaml
patches:
  - op: exec
    plugin: frontmatter-enhancer
    condition:
      frontmatter:
        draft: false
    params:
      addModifiedDate: true
```

## API Reference

### Plugin Interface

```typescript
interface Plugin {
  // Required fields
  name: string;              // Plugin identifier
  version: string;           // Semantic version
  apply: PluginFunction;     // Transformation function

  // Optional fields
  validate?: PluginValidateFunction;  // Parameter validation
  description?: string;      // Plugin description
  params?: PluginParamSchema[];  // Parameter schema
}
```

### PluginFunction

```typescript
type PluginFunction = (
  content: string,
  params: Record<string, unknown>,
  context: PluginContext,
) => string | Promise<string>;
```

### PluginValidateFunction

```typescript
type PluginValidateFunction = (
  params: Record<string, unknown>,
) => PluginValidationError[] | Promise<PluginValidationError[]>;
```

### PluginValidationError

```typescript
interface PluginValidationError {
  param: string;      // Parameter name
  message: string;    // Error message
  expected?: string;  // Expected value/type
  actual?: string;    // Actual value received
}
```

### PluginContext

```typescript
interface PluginContext {
  file: string;                    // Absolute file path
  config: Readonly<KustomarkConfig>;  // Full config
  env: Readonly<Record<string, string>>;  // Safe env vars
  relativePath: string;            // Relative output path
}
```

### PluginConfig

```typescript
interface PluginConfig {
  name: string;       // Plugin name (must match plugin export)
  source: string;     // File path or npm package
  version?: string;   // npm version constraint
  checksum?: string;  // SHA256 checksum
  timeout?: number;   // Timeout in milliseconds
}
```

## Best Practices

### 1. Keep Plugins Focused

Each plugin should do one thing well:

- Good: `toc-generator` - generates TOC
- Bad: `all-in-one` - TOC + word count + link validation

### 2. Provide Sensible Defaults

```javascript
export const apply = (content, params, context) => {
  const maxDepth = params.maxDepth ?? 3;  // Default to 3
  const title = params.title ?? 'Table of Contents';
  // ...
};
```

### 3. Validate Input Parameters

Always validate parameters, especially:

- Numeric ranges
- Enum values
- Required parameters
- Type constraints

```javascript
export const validate = (params) => {
  const errors = [];

  if (params.required === undefined) {
    errors.push({
      param: 'required',
      message: 'required parameter is required'
    });
  }

  return errors;
};
```

### 4. Handle Edge Cases

```javascript
export const apply = (content, params, context) => {
  // Handle empty content
  if (!content.trim()) {
    return content;
  }

  // Extract headers
  const headers = extractHeaders(content);

  // Skip if no headers found
  if (headers.length === 0) {
    return content;
  }

  // Process...
};
```

### 5. Use Descriptive Error Messages

```javascript
if (params.maxDepth < 1 || params.maxDepth > 6) {
  throw new Error(
    `maxDepth must be between 1 and 6, got ${params.maxDepth}`
  );
}
```

### 6. Document Your Plugin

Use JSDoc comments:

```javascript
/**
 * Table of Contents Generator
 *
 * Generates a table of contents from markdown headers.
 * Supports configurable depth and custom titles.
 *
 * @example
 * ```yaml
 * patches:
 *   - op: exec
 *     plugin: toc-generator
 *     params:
 *       maxDepth: 3
 * ```
 */
export const name = 'toc-generator';
```

### 7. Performance Considerations

- Avoid expensive operations in loops
- Cache computed values when possible
- Use regex efficiently
- Consider async for I/O operations

```javascript
// Bad: Recomputes on every iteration
for (const line of lines) {
  const headers = extractHeaders(content);  // Expensive!
}

// Good: Compute once
const headers = extractHeaders(content);
for (const line of lines) {
  // Use headers
}
```

### 8. Testing

Test your plugins thoroughly:

- Unit tests for transformation logic
- Edge cases (empty content, malformed input)
- Parameter validation
- Error handling

```javascript
// test/toc-generator.test.js
import { test, expect } from 'bun:test';
import { apply } from '../plugins/toc-generator.js';

test('generates TOC for valid headers', () => {
  const content = '# Header 1\n\n## Header 2';
  const result = apply(content, {}, mockContext);

  expect(result).toContain('Table of Contents');
  expect(result).toContain('[Header 1]');
});
```

Count words and add statistics:

```javascript

export const name = 'word-counter';
export const version = '1.0.0';

export const apply = (content, params, context) => {
  const words = content.trim().split(/\s+/).length;

};

## Troubleshooting

### Plugin Not Found

**Error**: `Plugin 'my-plugin' not found in plugin registry`

**Solutions**:

1. Check plugin is declared in `plugins` section
2. Verify plugin name matches the exported `name`
3. Check plugin source path is correct

```yaml
plugins:
  - name: my-plugin  # Must match export const name = 'my-plugin'
    source: ./plugins/my-plugin.js  # Check path exists
```

### Interface Validation Failed

**Error**: `Plugin 'my-plugin' does not implement the required interface`

**Solutions**:

1. Ensure plugin exports `name`, `version`, and `apply`
2. Check exports are at module level (not default export)
3. Verify function signatures match expected types

```javascript
// Correct
export const name = 'my-plugin';
export const version = '1.0.0';
export const apply = (content, params, context) => content;

// Wrong - default export
export default {
  name: 'my-plugin',
  version: '1.0.0',
  apply: (content, params, context) => content
};
```

### Plugin Timeout

**Error**: `Plugin 'my-plugin' exceeded timeout of 30000ms`

**Solutions**:

1. Optimize plugin performance
2. Increase timeout in config
3. Make long-running operations async

```yaml
plugins:
  - name: slow-plugin
    source: ./plugins/slow.js
    timeout: 60000  # Increase to 60 seconds
```

### Parameter Validation Failed

**Error**: `Plugin 'my-plugin' parameter validation failed`

**Solutions**:

1. Check parameter types match expected types
2. Provide all required parameters
3. Review plugin's `params` schema

```yaml
patches:
  - op: exec
    plugin: toc-generator
    params:
      maxDepth: 3  # Must be number between 1-6
      title: "Contents"  # Must be string
```

### Checksum Verification Failed

**Error**: `Plugin 'my-plugin' checksum verification failed`

**Solutions**:

1. Plugin file was modified - regenerate checksum
2. Wrong checksum in config - verify hash
3. Remove checksum if not needed

```bash
# Regenerate checksum
sha256sum plugins/my-plugin.js

# Update config with new hash
plugins:
  - name: my-plugin
    source: ./plugins/my-plugin.js
    checksum: sha256:new_hash_here
```

### Plugin Returns Wrong Type

**Error**: `Plugin returned number instead of string`

**Solutions**:

1. Ensure `apply` function returns a string
2. Convert values to strings before returning
3. Return original content if no changes

```javascript
// Wrong
export const apply = (content, params, context) => {
  return 12345;  // Not a string!
};

// Correct
export const apply = (content, params, context) => {
  return String(12345);  // Convert to string
};

// Best
export const apply = (content, params, context) => {
  return `${content}\n\nValue: 12345`;
};
```

### Import Errors

**Error**: `Cannot find module 'some-package'`

**Solutions**:

1. Install dependencies: `npm install some-package`
2. Use relative imports for local files
3. Check node_modules is accessible

```javascript
// If using external packages, install them first
import { parse } from 'yaml';  // Requires: npm install yaml

// For local utilities
import { helper } from './utils.js';  // Relative import
```

### Common Pitfalls

1. **Modifying context**: Context is readonly - don't try to modify it
2. **Side effects**: Avoid file I/O or global state changes
3. **Infinite loops**: Be careful with regex on untrusted input
4. **Memory leaks**: Don't accumulate data across invocations
5. **Error swallowing**: Don't catch and ignore errors silently
